Debugging

Recall our 3 main types of errors that we can get while programming they are static, static semantic and semantic. The first two 99% of the time the compiler will underline red for you while you are coding. You can then read what the problem is and fix it before running the program. The last kind however, is much more difficult to catch and fix. Semantic errors are when you write the code properly it runs fine however, your output from the code is not what you expected. This means there is a mistake in your logic. Example:
You are writing code to take persons name and change it to their first initial then a period then their last name. When you take in the name John Smith the program's output is J. mith even though we wanted J. Smith. This is an example of semantic errors, they happen during the programs run time. The code runs fine and the only way to fix them is read through your code line by line until you find the mistake in logic.

The Debugger

The debugger is used to run through your code step by step. In keeps track of your variables and checks your logical tests. There is a way to add in "breakpoints" by left clicking beside the line number. This will tell your code to stop at this location and freeze. From here you can read what all the current variables are set to, and then choose to "step over" or "step into". Step over will execute the line of code and evaluate all expressions and tell then pause again after the line has run. You can then go through and check how anything has changed. Step into will go in to any methods that are in the line of code so you can then go through that method step by step. It is advised that if it is your own built method then step into is a good idea. If it is a method provided by a library or java it self then assume that method is working correctly.

Other ways to debug

Some times you may want to just do a simple check while you are coding. One way to do this is add println statements to different parts of your code.
